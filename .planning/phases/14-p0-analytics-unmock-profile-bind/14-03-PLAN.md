<objective>
Remove `User.mock` from `ProfileViewModel` production code path. Bind the Profile tab's user data to the real current-session user.

Purpose: The Profile tab today shows fake data for every signed-in user because `ProfileViewModel.user` defaults to `User.mock` and is never replaced. The audit flagged this as a P0. The real source is `AuthManager.shared` (Firebase Auth), not `AccountRepository` — the audit's original phrasing was an oversimplification. `AccountRepository` exists but exposes sessions/login-history/exports/consents/creator-profile-update, not a current-user fetch. The fix is to hydrate `user` from `AuthManager.shared.currentUser` (or whatever the Firebase Auth-derived getter is) on `onAppear` and keep `User.mock` only as a Preview/test fallback.

Output: `ProfileViewModel` no longer holds `User.mock` in its production-reachable default. `ProfileView` calls a new `@MainActor loadProfile()` on `.task`/`.onAppear`. `User.mock` is retained only in `#if DEBUG` or in Preview providers. If `AuthManager` doesn't expose a ready-made user getter, add a thin `currentUser() -> User?` helper that maps Firebase's `User` type to our domain `User` model. Loading and error states handled visibly.
</objective>
